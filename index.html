windows下的异常处理

Posted on 2020-12-23 | In windows , 异常处理

基本概念

在应用程序执行过程中发生不常见事件。由CPU引发的叫做硬件异常。如：访问一个无效地址。由操作系统或应用程序引发的称作软件异常。

当有异常或中断发生时，windows会查找IDT(Interrupt Descriptor Table)来寻找处理函数。

IDT

IDT是一张位于物理内存中的线性表，共有256项，每项32位下每个长度8字节，64位下长度64字节。位置和长度由CPU中的IDTR（共有48位高32位是表的基址，低16位位表的长度，可以使用SIDT与LIDT来读写）寄存器决定

IDT的每一项为一个门结构，分为三类任务门（CPU任务切换）、中断门（中断处理程序入口）、陷阱门（异常处理入口）。

当有一个中断或异常发生时，CPU会根据终端号查找IDT，从而找到其处理程序入口地址。

准备活动

异常处理函数通常会对异常信息进行封装，以便进行后续信息，一般会封装为两个结构体一部分为EXCEPTION_RECORD用于记录异常相关的信息，另一部分使用KTRAP_FRAME进行封装，记录了异常发生时CPU各寄存器的完整镜像，但这是一个内核态下的概念要在用户态中使用将会被封装为CONTEXT即上下文结构体。

包装完毕后，异常处理函数还会进一步调用系统内核的nt!KiDispatchException函数来处理异常，会根据错误发生时的系统状态不同而进行不同的处理。

内核态错误的处理过程

  1. 检查是否存在内核调试器，若存在系统把异常处理的权力交给内核调试器，并注明是第一次处理机会，交给内核调试器后会根据用用户的设置来确定是否需要处理该异常，若无法判断则中断，或正确处理则返回发生异常的位置继续执行。若不存在则跳过。
  2. 若不存在内核调试器或在内核调试器中选择不进行处理，则会调用nt!RtlDisapatchException由线程所定义的SEH结构体进行处理。
  3. 若此时仍未处理，则调试器会对其进行第二次处理。
  4. 若此时仍然不处理，系统则认为无法再继续运行下去了，为了避免引起更严重的错误，系统此时将调用KeBugCheckEx产生一个KERNEL_MODE_EXCEPTION_NOT_HANDLED的蓝屏错误。

用户态错误的处理过程

  1. 首先检查内核调试器是否存在，若存在则将控制权交给内核调试器，给予第一次异常处理机会。
  2. 若不存在或不处理该错误，则会在栈上放置EXCEPTION_RECORD与CONTEXT两个结构，并返回用户态的KiUserExceptionDispatcher函数，由他调用ntdll!RtlDispatchException函数进行用户态的异常处理，此时涉及SEH与VEH两种异常处理机制，若此时调试存在则会跳过SEH部分的顶级异常处理，进入下一阶段的处理。否则进入顶级异常处理阶段，进行终结处理，如果没有调试器或者调试器还是处理不了异常，则系统就会调用ExitProcess函数。
  3. 若不能处理则给予调试器第二次处理的机会，若还不处理就会将异常传送给异常处理端口进行处理，该端口由子系统进程csrss.exe监听，通常会弹出一对话框，选择结束程序或者将其附加至调试器上，若没有调试器或无法处理异常，则调用ExitProcess函数结束。
  4. 在终结程序时，会再次调用线程中的所有异常处理过程，这是最后的机会，否则将会被清理。

在浏览器中document.documentElement.clientWidth，document.documentElement.clientHeight中保存了页面的视口信息。

补充:作用域链

每个函数都有自己的执行环境。当执行流进入一个函数时，函数的环境会被推入一个环境栈，函数执行完成之后，栈将执行环境推出，并把控制权交还原来的执行环境。

当代码在一个环境中执行时，会创建一个变量对象的作用域链(scope chain)以保证对执行环境有权访问的所有变量和函数的有序访问。如果这个环境是函数，则将其活动对象作为变量对象。

活动对象：最初只包含arguments对象。作用域链的下一个变量对象来自包含环境。探索过程始终从作用于的前端开始，然后逐级向后回溯。

内部环境可以通过作用域访问所有的外部环境，而外部环境不能访问内部环境的值。

延长作用域链（两种方式 try-catch语句/with语句）：

对catch语句来说，会创建一个新的变量对象，其中包含被抛出的错误对象的声明。

对with语句来说，会将指定的对象添加到作用域链之中。

没有块级作用域： if没有独立的执行环境，退出if代码块之后，其中声明的变量会被添加到外部作用域之中。for语句中的变量循环结束后仍然会被存放在当前的执行环境中。

通过var声明的变量会被添加到最近的环境之中，没有使用var关键词声明的变量会直接添加到全局作用于之中。

标识符的搜索：会从作用域链的前端开始，向上搜索，若找到则搜索停止。

IA-32寄存器

IA-32的寄存器主要可以分为4类。

  1. 通用寄存器
  2. 段寄存器
  3. 程序状态与控制寄存器
  4. 指令指针寄存器

通用寄存器

通用性寄存器，用于传送和暂存数据，参与算术运算，保存运算结果。

EAX：累加器(针对操作数和结果数据)

EBX：基质寄存器(DS段中的数据指针)

ECX：计数器(字符串和循环操作)

EDX：数据寄存器(I/O指针)

所有win32API函数都会把API保存至EAX再返回。

WIN32 API会再内部使用ECX和EDX，调用前需保护现场。

EBP: 扩展基址指针寄存器

ESI: 源变址寄存器

EDI: 目的变址寄存器

ESP: 栈指针寄存器

浅拷贝与深拷贝

深浅拷贝只针对Object与Array等应用类型的拷贝。

浅拷贝：只复制某个对象的指针，而不复制对象本身，新旧对象共享一块内存。

深拷贝：复制某个对象产生一个相同的拷贝。
